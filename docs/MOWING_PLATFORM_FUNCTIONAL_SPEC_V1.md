# 割草服务平台功能说明书 V1

## 1. 总述

本平台用于连接草坪服务用户、个人割草工和平台运营方。平台初期的核心不是做一个完全自动化的机器人割草系统，而是把“机器人完成大面积重复割草”和“割草工完成现场精细收尾”组合成稳定可交付的服务。

第一期目标是跑通真实业务闭环：

1. 用户可以提交割草服务需求。
2. 平台可以审核、定价、派单和跟踪服务。
3. 割草工可以接单、到场、使用机器人完成主体割草，并人工处理边缘、树下、墙边、角落等机器人难以处理的区域。
4. 平台可以通过照片、状态和异常记录控制服务质量。
5. 平台可以轻量记录机器人状态，但不把机器人维护作为割草工第一期的主要任务。

第一期平台应优先服务运营可控性，而不是追求高度自动化。报价、派单、异常处理和质量审核可以先由平台人工完成。

## 2. 分项功能

### 2.1 用户端功能

#### 2.1.1 用户注册与登录

用户可以通过手机号、邮箱或第三方账号注册和登录。第一期只需要支持基础身份识别，不需要复杂会员体系。

主要功能：

- 创建用户账号。
- 维护姓名、电话、邮箱等基础信息。
- 管理一个或多个服务地址。
- 查看历史订单。

#### 2.1.2 服务地址与草坪信息

用户下单时需要提供足够的现场信息，帮助平台判断价格、时间和是否适合机器人辅助作业。

主要字段：

- 服务地址。
- 草坪大概面积或范围。
- 草坪照片。
- 草坪入口说明。
- 是否有宠物。
- 是否有斜坡、树木、花坛、栅栏、障碍物。
- 是否有机器人不适合作业的区域。
- 用户备注。

#### 2.1.3 下单功能

用户可以提交一次性割草或定期割草需求。

第一期服务类型：

- 一次性割草。
- 定期割草预约。

下单流程：

1. 选择服务地址。
2. 上传草坪照片。
3. 填写草坪和现场情况。
4. 选择期望服务日期和时间段。
5. 提交订单。
6. 等待平台确认价格和服务安排。

第一期可以采用“提交需求后平台确认报价”的方式，不强制做全自动报价。

#### 2.1.4 订单状态查看

用户可以查看订单从提交到完成的状态。

建议状态：

- 待确认。
- 已报价。
- 待支付或已授权支付。
- 已派单。
- 割草工已接单。
- 割草工已到达。
- 服务中。
- 待平台确认。
- 已完成。
- 已取消。
- 异常处理中。

#### 2.1.5 服务结果确认

用户可以查看服务前后照片，并对服务进行评价。

主要功能：

- 查看服务前照片。
- 查看服务后照片。
- 确认服务完成。
- 提交评分。
- 提交投诉或补充说明。

### 2.2 割草工端功能

#### 2.2.1 割草工注册与审核

割草工第一期主要是个人服务者，需要经过平台审核后才能接单。

主要字段：

- 姓名。
- 手机号。
- 服务区域。
- 可服务时间。
- 基础经验说明。
- 交通方式。
- 可携带工具说明。
- 平台审核状态。

#### 2.2.2 接单与订单查看

割草工可以查看分配给自己的订单信息，并决定是否接单。

可见信息：

- 服务地址。
- 预约时间。
- 草坪照片。
- 用户备注。
- 平台服务要求。
- 是否需要携带机器人。
- 是否有特殊现场风险。
- 预计收入。

第一期建议以平台派单为主，割草工确认接单为辅，暂不做公开抢单市场。

#### 2.2.3 到场与服务流程

割草工的核心职责是现场交付，不是机器人维修。

标准服务步骤：

1. 到达现场并打卡。
2. 上传服务前照片。
3. 检查草坪入口和明显障碍物。
4. 使用机器人完成主体草坪割草。
5. 人工处理机器人难以完成的区域。
6. 清理明显草屑和现场遗留问题。
7. 上传服务后照片。
8. 填写完成检查项。
9. 提交完成。

#### 2.2.4 精细收尾检查项

割草工需要明确完成机器人无法很好处理的区域。

建议检查项：

- 主草坪已完成割草。
- 草坪边缘已修整。
- 树下已处理。
- 墙边已处理。
- 栅栏边已处理。
- 花坛边已处理。
- 障碍物周围已处理。
- 角落区域已处理。
- 小块机器人无法进入区域已处理。
- 现场已简单整理。
- 服务后照片已上传。

这些检查项是第一期服务质量的核心。

#### 2.2.5 异常上报

割草工可以在现场上报异常，避免服务质量和结算争议。

异常类型：

- 下雨或草坪过湿。
- 草太高，需要额外处理。
- 入口无法进入。
- 用户信息不准确。
- 现场面积明显超出描述。
- 地形不适合机器人。
- 机器人无法正常工作。
- 用户临时提出超范围要求。
- 存在安全风险。

异常上报应包含照片、文字说明和建议处理方式。

#### 2.2.6 收入与结算

割草工可以查看自己的服务收入。

主要功能：

- 查看已完成订单。
- 查看每单预计收入。
- 查看平台确认状态。
- 查看待结算金额。
- 查看已结算金额。

第一期可以先做后台人工结算记录，不必接入复杂自动分账。

### 2.3 平台后台功能

#### 2.3.1 订单管理

平台后台是第一期运营核心。

主要功能：

- 查看全部订单。
- 筛选订单状态。
- 查看用户提交的草坪信息和照片。
- 人工确认价格。
- 修改服务时间。
- 手动派单。
- 改派订单。
- 标记异常处理结果。
- 审核服务完成。

#### 2.3.2 价格管理

第一期不需要完全自动报价，但需要沉淀价格规则。

可配置因素：

- 草坪面积。
- 服务距离。
- 草况复杂度。
- 边缘和障碍物复杂度。
- 是否定期服务。
- 是否加急。
- 是否存在明显超范围工作。

价格可以先由平台运营人员确认，系统记录价格构成。

#### 2.3.3 割草工管理

平台需要管理个人割草工的准入和表现。

主要功能：

- 审核割草工。
- 设置服务区域。
- 查看接单记录。
- 查看完成率。
- 查看取消率。
- 查看评分。
- 查看异常记录。
- 暂停或恢复接单资格。

#### 2.3.4 服务质量审核

平台通过照片和检查项确认服务质量。

主要功能：

- 查看服务前后照片。
- 查看割草工检查项。
- 查看异常上报。
- 标记通过、驳回或需要补充说明。
- 触发客户沟通或返工流程。

#### 2.3.5 机器人轻量管理

机器人是平台生产工具，第一期只需要管理可用性和异常，不需要完整维修系统。

主要功能：

- 机器人编号。
- 当前状态：可用、使用中、故障、需要检查、维修中。
- 当前关联割草工或订单。
- 最近使用时间。
- 最近异常记录。
- 备注。

割草工第一期只需要反馈机器人是否可用，以及上报明显异常。换刀片、保养计划、配件库存和维修工单可放到第二期。

#### 2.3.6 投诉与异常处理

平台需要集中处理影响交付的事件。

主要功能：

- 查看用户投诉。
- 查看割草工异常上报。
- 记录平台处理结果。
- 触发退款、补割、改派或关闭订单。

### 2.4 机器人相关边界

第一期机器人能力在平台中只作为辅助服务能力体现。

机器人负责：

- 大面积、规则区域的主体割草。
- 提高割草工单位时间服务效率。

割草工负责：

- 现场服务交付。
- 机器人无法覆盖区域的人工修整。
- 服务前后照片。
- 现场异常判断。
- 基础使用反馈。

平台负责：

- 机器人资产归属和状态管理。
- 服务标准制定。
- 质量审核。
- 异常处理。

第一期不要求割草工承担机器人维修责任。

## 3. 总结

第一期平台应围绕“用户下单、平台派单、割草工现场交付、机器人辅助主体割草、人工精细收尾、平台质量审核”构建。平台真正交付给用户的不是单纯机器人割草，而是一套更稳定的草坪服务：机器人提高效率，割草工保证细节，平台保证标准。

建议第一期只做必要功能，避免过早建设自动派单、自动报价、复杂维修、开放市场和机器人深度远程控制。等订单量、服务标准和机器人现场表现稳定后，再进入第二期。
