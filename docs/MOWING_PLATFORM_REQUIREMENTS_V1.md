# 割草服务平台需求文档 V1

## 1. 总述

### 1.1 背景

平台计划连接有割草需求的用户、提供现场服务的个人割草工，以及拥有或运营割草机器人的平台方。机器人可以显著提高主体草坪割草效率，但在边缘、树下、墙边、栅栏边、花坛边、障碍物周围和角落等区域仍需要人工处理。

因此，第一期产品不应定位为“全自动机器人割草平台”，而应定位为“机器人辅助的专业割草服务平台”。

### 1.2 产品目标

第一期目标是验证业务闭环和服务标准：

- 用户能够提交真实割草需求。
- 平台能够确认订单、报价、派单和审核质量。
- 割草工能够使用机器人完成主体割草，并人工完成精细收尾。
- 平台能够记录机器人基础状态和异常。
- 订单能够完成结算和评价。

### 1.3 第一阶段成功标准

第一期上线后，应能够判断：

- 用户是否愿意通过平台下单。
- 割草工是否能按平台标准完成服务。
- 机器人是否能提升现场服务效率。
- 平台是否能控制质量、异常和成本。
- 该服务模式是否具备继续规模化的价值。

## 2. 分项需求

### 2.1 用户角色

#### 2.1.1 普通用户

普通用户是有草坪割草需求的个人或家庭用户。

核心需求：

- 快速提交割草需求。
- 清楚知道服务价格、时间和状态。
- 获得完整的割草服务，而不只是机器人能覆盖的部分。
- 可以查看服务前后照片。
- 可以评价、投诉或要求平台处理问题。

#### 2.1.2 割草工

割草工是第一期服务交付主体，通常是个人服务者。

核心需求：

- 查看平台分配的订单。
- 知道服务范围和现场要求。
- 使用机器人提高主体割草效率。
- 完成边缘、树下、角落等人工精细收尾。
- 上传服务记录。
- 上报异常。
- 查看收入和结算。

#### 2.1.3 平台运营人员

平台运营人员负责业务调度和质量控制。

核心需求：

- 审核订单和报价。
- 派单给合适割草工。
- 管理割草工和服务区域。
- 查看机器人基础状态。
- 审核服务质量。
- 处理异常、投诉、退款和补割。

### 2.2 用户端需求

#### R-U-001 用户注册登录

用户应能够注册和登录平台。

验收标准：

- 用户可以创建账号。
- 用户可以维护基础联系方式。
- 用户可以查看自己的订单。

优先级：P0

#### R-U-002 服务地址管理

用户应能够填写服务地址和现场说明。

验收标准：

- 用户可以保存服务地址。
- 用户可以填写入口、宠物、障碍物等说明。
- 用户可以上传草坪照片。

优先级：P0

#### R-U-003 提交割草订单

用户应能够提交一次性或定期割草需求。

验收标准：

- 用户可以选择服务类型。
- 用户可以选择期望服务日期和时间段。
- 用户可以提交草坪照片和备注。
- 系统生成订单并进入待确认状态。

优先级：P0

#### R-U-004 查看订单状态

用户应能够查看订单处理进度。

验收标准：

- 用户可以看到订单当前状态。
- 状态变化对用户可见。
- 用户可以看到平台确认后的价格和服务时间。

优先级：P0

#### R-U-005 查看服务结果

用户应能够查看服务前后照片并评价服务。

验收标准：

- 用户可以查看服务前照片。
- 用户可以查看服务后照片。
- 用户可以提交评分和反馈。

优先级：P0

#### R-U-006 投诉与问题反馈

用户应能够对服务问题发起反馈。

验收标准：

- 用户可以选择问题类型。
- 用户可以提交文字和照片。
- 平台可以在后台查看并处理。

优先级：P1

### 2.3 割草工端需求

#### R-W-001 割草工注册申请

割草工应能够提交加入申请。

验收标准：

- 割草工可以填写姓名、联系方式、服务区域和可服务时间。
- 平台可以审核通过或拒绝。
- 未审核通过的割草工不能接单。

优先级：P0

#### R-W-002 查看和确认订单

割草工应能够查看平台分配的订单并确认是否接单。

验收标准：

- 割草工可以看到订单地址、时间、照片和备注。
- 割草工可以接单或拒单。
- 平台可以看到割草工响应结果。

优先级：P0

#### R-W-003 到场打卡

割草工应能够在到达现场后标记到达。

验收标准：

- 割草工可以标记已到达。
- 系统记录到达时间。
- 订单状态同步更新。

优先级：P0

#### R-W-004 服务前后照片

割草工应上传服务前和服务后照片。

验收标准：

- 服务开始前必须上传至少一张照片。
- 服务完成后必须上传至少一张照片。
- 平台后台可以查看照片。

优先级：P0

#### R-W-005 精细收尾检查

割草工应按平台标准完成机器人无法覆盖区域的人工处理。

验收标准：

- 割草工必须勾选主草坪、边缘、树下、墙边、栅栏边、花坛边、障碍物周围、角落、现场整理等检查项。
- 未完成必要检查项时不能提交完成。
- 平台可以查看检查项结果。

优先级：P0

#### R-W-006 异常上报

割草工应能在现场上报影响服务的问题。

验收标准：

- 割草工可以选择异常类型。
- 割草工可以上传照片和文字说明。
- 异常提交后订单进入异常处理中。

优先级：P0

#### R-W-007 收入查看

割草工应能查看订单收入和结算状态。

验收标准：

- 割草工可以查看已完成订单收入。
- 割草工可以查看待结算和已结算金额。

优先级：P1

### 2.4 平台后台需求

#### R-A-001 订单管理

平台应能够统一管理所有订单。

验收标准：

- 平台可以查看订单列表和详情。
- 平台可以筛选订单状态。
- 平台可以修改服务时间、价格和备注。
- 平台可以取消或关闭订单。

优先级：P0

#### R-A-002 人工报价

平台应能够基于用户提交信息确认服务价格。

验收标准：

- 平台可以录入最终价格。
- 平台可以记录价格备注。
- 用户可以看到平台确认后的价格。

优先级：P0

#### R-A-003 手动派单

平台应能够把订单分配给割草工。

验收标准：

- 平台可以选择割草工派单。
- 派单后割草工可以收到订单。
- 平台可以改派订单。

优先级：P0

#### R-A-004 割草工管理

平台应能够审核和管理割草工。

验收标准：

- 平台可以审核申请。
- 平台可以查看服务记录、评分和异常。
- 平台可以暂停割草工接单资格。

优先级：P0

#### R-A-005 服务质量审核

平台应能够审核服务是否完成。

验收标准：

- 平台可以查看服务前后照片。
- 平台可以查看检查项和异常记录。
- 平台可以确认完成、驳回或要求补充说明。

优先级：P0

#### R-A-006 机器人基础状态管理

平台应能够记录机器人基本可用性。

验收标准：

- 平台可以创建机器人编号。
- 平台可以设置机器人状态。
- 平台可以关联机器人与订单或割草工。
- 平台可以查看最近异常和备注。

优先级：P1

#### R-A-007 投诉和异常处理

平台应能够处理用户投诉和割草工异常。

验收标准：

- 平台可以查看异常列表。
- 平台可以记录处理结果。
- 平台可以触发退款、补割、改派或关闭订单。

优先级：P1

### 2.5 非功能需求

#### 2.5.1 可运营性

第一期应优先保证平台运营人员能人工介入关键流程。

要求：

- 报价可人工确认。
- 派单可人工操作。
- 异常可人工处理。
- 质量可人工审核。

#### 2.5.2 可追溯性

关键动作应保留记录。

要求：

- 订单状态变化有时间记录。
- 割草工到场和完成有时间记录。
- 服务照片可追溯。
- 异常和投诉有处理记录。

#### 2.5.3 安全边界

第一期不做机器人远程运动控制功能。如果后续增加远程控制，必须单独设计权限、确认、日志、超时、失败状态和安全停止机制。

#### 2.5.4 隐私

用户地址、联系方式、照片和订单信息属于敏感数据，应限制访问范围。

### 2.6 第一期不包含内容

以下内容不进入第一期：

- 自动派单算法。
- 完全自动报价。
- 开放抢单市场。
- 割草工等级体系。
- 复杂机器人维修工单。
- 刀片和配件库存管理。
- 机器人远程控制。
- 地图画草坪边界。
- 自动估算草坪面积。
- 保险和复杂合同系统。
- 自动分账。

## 3. 总结

第一期需求应聚焦“服务闭环”而不是“技术炫技”。用户端解决下单和结果查看，割草工端解决现场执行和精细收尾，平台后台解决报价、派单、质量审核和异常处理。机器人只做轻量状态记录，不把维护职责压给割草工。

该需求范围足以支撑早期真实运营，并为后续自动报价、自动派单、机器人维护、面积识别和订阅套餐等第二期功能留下扩展空间。
