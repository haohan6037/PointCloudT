# 割草服务平台阶段 1 执行清单 V1

## 1. 总述

阶段 1 的目标是先完成“平台后台和订单基础”。这一阶段不追求完整用户端和割草工端，而是先让平台运营人员可以管理订单、人工报价、手动派单，并掌握用户、服务地址、草坪信息和割草工基础档案。

阶段 1 成功后，平台应具备内部试运营能力：运营人员可以人工录入或接收一笔订单，确认价格，分配割草工，并跟踪订单状态。

## 2. 执行范围

### 2.1 本阶段包含

- 后台订单列表。
- 后台订单详情。
- 订单状态流转。
- 人工报价。
- 手动派单。
- 用户基础信息。
- 服务地址和草坪信息。
- 割草工基础档案。
- 演示数据。
- 阶段 1 验收检查。

### 2.2 本阶段不包含

- 用户完整下单端。
- 割草工移动端执行流程。
- 照片真实上传存储。
- 支付和结算。
- 机器人管理页面。
- 投诉和异常处理完整流程。
- 自动报价。
- 自动派单。
- 机器人远程控制。

## 3. 任务拆分

验收标准：

- quoted：已报价。

- in_service。

验收标准：

- 后台订单列表能按状态筛选。
- 订单详情能展示当前状态。
- 报价和派单操作能推动状态变化。

执行要求：

## 4. 建议技术执行路径

当前仓库已有 `index.html`，但它是 PointCloudTT 点云工作台，不建议直接覆盖。

推荐两种路径：

### 路径 A：静态后台原型

适合目标：快速给领导和运营人员确认流程。

做法：

- 新建 `mowing-platform/admin-prototype.html`。
- 使用本地静态演示数据。
- 实现订单列表、订单详情、报价和派单交互。

优点：

- 最快。
- 不需要后端。
- 适合汇报和流程确认。

缺点：

- 不能真实存储数据。
- 后续需要迁移到正式应用。

### 路径 B：独立 Web 应用骨架

适合目标：直接进入可扩展产品开发。

做法：

- 新建 `mowing-platform/` 应用目录。
- 建立前端页面、数据模型和本地 mock API。
- 后续逐步接入真实后端和数据库。

优点：

- 更接近正式产品。
- 后续扩展用户端、割草工端更顺。

缺点：

- 启动成本高于静态原型。

## 5. 下一步执行建议

建议先走路径 A，用 1 个静态后台原型把阶段 1 跑通。等领导确认流程和字段后，再决定是否进入路径 B。

下一步最小任务：

1. 新建 `mowing-platform/admin-prototype.html`。
2. 放入阶段 1 演示数据。
3. 做订单列表、订单详情、人工报价、手动派单。
4. 用浏览器验证状态流转。

这样可以最快把“第一阶段开始执行”变成可看的产品雏形。
